# OpenCode Agent Instructions — Space Sorcerers Docs

## Project Overview

Репозиторий документации Space Station 14 и Robust Toolbox, работающий на **Mintlify**.  
Файлы — `.mdx`, структура — `en/` (английский) и `ru/` (русский, в процессе перевода).  
CI: `mintlify validate` на PR, `mintlify export` → GitHub Pages на push в `master`.

---

## Основные правила работы

### Запрещено
- Менять `docs.json` без явной просьбы
- Создавать новые `.mdx` страницы без обсуждения
- Менять URL-структуру, редиректы, навигацию
- Использовать HTML-комментарии (`<!-- -->`) — только `{/* */}`
- Оставлять `style="..."` (строки) в JSX — только `style={{...}}` (объект)

### Перевод статей (en → ru)
Используй промпт из `ai/translate-prompt.md` как руководство.  
Ключевые принципы:
- Не переводить: код, команды Git, API/классы движка, URL, имена собственные
- YAML frontmatter: ключи не трогать, `title`/`description` — переводить
- `<div style="...">` → `<div style={{...}}>` при необходимости
- Комментарии: `<!-- -->` → `{/* */}`

### CI/CD
- PR → `test.yml` (mintlify validate + Python тесты)
- push master → `deploy.yml` (mintlify export + GitHub Pages)

### Приоритеты правил (при конфликте применяй первое):
1. **Работоспособность MDX** — синтаксис, код, идентификаторы, ссылки, компоненты не должны быть сломаны.
2. **Сохранение кода и технических идентификаторов** — всё, что ломает сборку или работу системы, переводить нельзя.
3. **Сохранение терминологии** — специфичные имена, аббревиатуры, устоявшиеся англицизмы из IT/Git/SS14 не переводятся.
4. **Качество перевода** — только после выполнения первых трёх пунктов переводи естественный текст на литературный технический русский.

#### Что НИКОГДА не переводить (оставляй латиницей)

**Имена собственные и названия**
- Названия статей, гайдов, проектов, инструментов, технологий, официальных документов, даже если они выглядят как обычные слова: *The Robust Book, Git for the SS14 Developer, Robust Toolbox, Space Station 14, Fork, TortoiseGit*, и т.п.
- Аббревиатуры: *SS14, PR, GUI, CLI, YAML, ECS, NPC, UI, IoC, HTTP, ACZ, MIDI, CBT, WIP, PVS*, и любые другие.
- Имена конкретных классов, компонентов, систем, типов, структур движка (даже без обратных кавычек): *EntitySystem, PrototypeManager, ViewVariables, FastSerializer, NetEntity, EntityUid, MapGrid, ContentManifest* — все подобные.
- Названия таблиц, enum, конфигурационных ключей, параметров, пространств имён.

**Git‑термины и команды**
- **Всегда на английском**: *branch, commit, merge, push, pull, fetch, fork, remote, submodule, staging, checkout, revert, rebase, force push, HEAD, upstream, origin, master, clone, log, diff, tag, working tree*, и любые другие термины Git.
- Команды полностью (даже если в тексте без кавычек): `git fetch upstream`, `python RUN_THIS.py`, `cd RobustToolbox`, `git add -A`.
- Аргументы и флаги: `--force`, `--oneline`, `--hard`.
- Имена веток, remote, путей, файлов: *master, upstream, funny-feature, .git/config, RUN_THIS.py, BuildChecker/*.

**Технические идентификаторы и код**
- **YAML frontmatter** (секция `---`) — не изменяй **ключи** и все служебные поля: `id, slug, sidebar_position, tags, keywords, draft, hide_title`.  
  **Разрешено переводить только значения полей `title` и `description`**, так как они отображаются пользователю в локализованной документации.
- Код в обратных кавычках (`` ` ``) и блоках ` ``` ` — полностью без изменений.
- JSX/HTML‑компоненты: не трогай имена тегов, атрибуты `id, value, className, key, slug, name`. Текстовые атрибуты, предназначенные для показа пользователю (`label, title, alt`), переводи, если они не являются именами собственными (пример: `label="Linux"` — оставить, `label="Installation"` — перевести). Содержимое между открывающим и закрывающим тегом переводи как обычный текст.
- URL в ссылках никогда не меняй. Текст ссылки переводи, только если это естественная фраза, а не название ресурса.
- Анкоры, хеши коммитов, GitHub/Discord‑имена, названия скриншотов, пути к изображениям не изменяй.
- Структура таблиц (количество столбцов, выравнивание) неприкосновенна.

#### Что можно и нужно переводить

**Естественный текст**
- Пояснения, инструкции, описания, советы.  
  Стиль — точный технический русский, без разговорных оборотов.

**Заголовки разделов**
- **Не переводи**, если заголовок — имя собственное (проект, гайд, технология).  
  **Переводи**, если заголовок чисто описательный.  
  Пример:  
  `# Git for the SS14 Developer` → оставить.  
  `# Setting up your repositories` → `Настройка repository` (repository оставляем как термин).  
  `## Branching & Commits` → `Branching и Commits`.

**Общие технические понятия**
Разрешён перевод устоявшихся русскоязычных аналогов для базовых терминов:
- entity → сущность
- component → компонент
- system → система
- repository → репозиторий
- network → сеть
- client → клиент
- server → сервер
- engine → движок
- tool → инструмент
- guide → руководство
- feature → функция / возможность (по контексту)

**Но!** Если такое слово — часть имени собственного класса/системы (например `EntitySystem`, `ComponentFactory`), оно **не переводится**.

#### Специальные инструкции

**Изображения**
```md
![Git history window](history.png)
```
Переводи текст описания, путь к файлу не трогай.

**Ссылки**
```md
[The Git-SCM online book](https://git-scm.com/book)
```
Если текст ссылки — название ресурса, оставляй как есть. Если обычная фраза («Click here»), переводи.

**Frontmatter**
```yaml
---
title: Git for the SS14 Developer
description: Learn how to use Git
sidebar_position: 5
---
```
Переведи `title` и `description`. Ключи и остальные поля не трогай.

**Inline HTML**
```html
<span>Click here</span>
```
Переводи только текст между тегами, теги и атрибуты сохраняй.

**Аббревиатуры в тексте**
Если аббревиатура расшифровывается, расшифровку можно перевести, но саму аббревиатуру оставляй:
`PVS (Potentially Visible Set)` → `PVS (потенциально видимый набор)`.

**Списки терминов**
Если встретил слово, которого нет в правилах, руководствуйся принципом: **если сомневаешься, является ли оно названием системы/класса/API — не переводи**.

#### Консистентность
Если термин один раз оставлен на английском, он должен оставаться английским во всём файле и во всех файлах перевода. Не смешивай «maintainer» и «мейнтейнер». Если разрешённый к переводу термин (entity) переведён как «сущность», используй только «сущность» для entity везде.

- [ ] Не изменены команды, флаги, пути, имена веток.
- [ ] Не изменены URL и анкоры.
