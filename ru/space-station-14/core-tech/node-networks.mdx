`Node`, `NodeContainer` и `INodeGroup` — это троица частей, составляющих 'слой связности' в SS14. Это используется для труб и проводов (энергии).

Содержание этого документа должно дать общий обзор системы.

## `NodeContainer` : Компонент в сущностях, взаимодействующих с системой узлов

`NodeContainer` — это компонент, с которым будет взаимодействовать устройство, которое вы создаёте, для доступа к системе узлов.

`Content.Server.GameObjects.Components.Atmos.Piping.Pumps.BasePumpComponent` следует рассматривать как 'референсный код' для компонента, которому нужно использовать узлы не слишком сложным способом, но требующим обдумывания, так как он не является 'универсально подключённым', как устройство WireNet.

## Подклассы `Node` : Определяют связность между собой и другими узлами

`Node` — это конкретная точка подключения на сущности.

Экземпляр `Node` — это конкретная точка подключения, и поведение этого экземпляра определяет, как эта точка подключения соединяется с другими узлами.

Например, каждая сторона насоса — это отдельный `PipeNode`, который подключается только в этом направлении, но обе стороны трубы — это один `PipeNode`, который подключается в обоих направлениях.

Но `AdjacentNode` не имеет способа определить такое направление и просто подключается ко всем соседним узлам того же типа.

Все `Node` имеют свойство `NodeGroupID`, доступное в прототипе как `nodeGroupID`, которое определяет совместимость между `Node` и типом `INodeGroup`, который используется.

Реализация `GetReachableNodes` управляет обнаружением `Node`, в то время как вызов `RefreshNodeGroup` — это способ вручную принудительно обновить этот узел.

Обратите внимание, что код `AdjacentNode` на самом деле не проверяет `NodeGroupID` — эта проверка выполняется базовым классом `Node` в `GetReachableCompatibleNodes`. Это подразумевает, что `GetReachableNodes` в общем случае не нужно выполнять эту проверку, поэтому выполняйте только необходимые проверки, как в случае с `PipeNode`, который может подключаться только к другим `PipeNode`.

Я не уверен, вызовет ли асимметричное подключение проблемы на практике, поэтому старайтесь не создавать ситуацию, где это может произойти (т.е. что-то, ведущее себя как `PipeNode` рядом с `AdjacentNode` — `AdjacentNode` подключится к `PipeNode`, но не наоборот).

Чтобы фактически использовать `Node` для чего-то полезного, `Node` должен либо получать информацию о своей родительской сети через `INodeGroup`, который имеет переопределяемые функции для добавления и удаления узлов, либо тот, кто к нему обращается, должен использовать его свойство `NodeGroup`, которое не имеет колбэков обновления и всегда имеет тип `INodeGroup`.

Хотя это уже должно быть ясно: `Node` внутри одной сущности не подключаются автоматически друг к другу, если это не принудительно обеспечено их поведением подключения (прямо или косвенно, например, через подключение насоса к самому себе).

Наконец, опять же, если вам нужно поведение 'подключиться к соседним сущностям той же группы', используйте `AdjacentNode`, как показано ниже:

```yml
  - type: NodeContainer
    nodes:
    - !type:AdjacentNode
      nodeGroupID: Apc
    - !type:AdjacentNode
      nodeGroupID: WireNet
```

## `INodeGroup` : Представляет подключённую группу `Node` и содержит поведение группы

Экземпляры `INodeGroup` представляют подключённую группу `Node`.

Это подразумевает, что любой `Node` в группе подключён к любому другому `Node`. (Не обязательно напрямую, но не проходя через какие-либо 'явные разделители', такие как насосы в системе PipeNet.)

Реализация `INodeGroup` выбирается через `NodeGroupID` узла. Смотрите `NodeGroupFactory.cs` (описано ниже) для подробностей.

Обратите внимание, что все реализации `INodeGroup` в настоящее время расширяют `BaseNodeGroup`.

## `NodeGroupFactory.cs` : Создаёт `INodeGroup` для `Node` и содержит enum `NodeGroupID`

`NodeGroupFactory.cs` содержит типы сетей в enum `NodeGroupID`.

`NodeGroupID` сопоставляются с реализациями `INodeGroup` через NodeGroup reflection attribute (такой как `[NodeGroup(NodeGroupID.Pipe)]` в `IPipeNet.cs`).

Несколько `NodeGroupID` могут быть сопоставлены одной реализации `NodeGroup`.

`NodeGroupFactory` (доступна через IoC как INodeGroupFactory) имеет один важный член: `MakeNodeGroup`. (Также есть `Initialize`, который вызывается из EntryPoint сервера и используется для сканирования классов с помощью reflection.)

`MakeNodeGroup` создаёт новый `INodeGroup` из `Node`, получая тип из `Node.NodeGroupID`.

## Приложение: Некоторые Power Group ID:

+ `Apc`: Кабели расширения APC -> `ApcNetNodeGroup`
+ `MVPower`: MV провод -> `PowerNetNodeGroup`
+ `HVPower`: HV провод -> `PowerNetNodeGroup`
